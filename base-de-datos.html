2. Estructuración de Datos: Llaves y Diagramas en SQL

30 de mayo de 2026 | Gestión de Archivos

Para diseñar una base de datos estable, no basta con crear tablas al azar. Primero debemos dibujar un diagrama de flujo o entidad-relación que sirva como mapa visual, y luego definir los identificadores únicos que conectarán toda nuestra información.

Llaves Primarias y Secundarias (Foráneas)

En el estándar de las bases de datos relacionales, las tablas necesitan comunicarse entre sí de forma ordenada. Para lograrlo sin errores ni datos duplicados, utilizamos dos tipos esenciales de identificadores:

  • Llave Primaria (Primary Key - PK): Es el campo que identifica de forma **única e irrepetible** a cada fila de una tabla. No puede haber dos registros con la misma llave primaria, ni tampoco puede quedarse vacía. Un ejemplo perfecto es el número de Carnet de un estudiante o el CUI.
  • Llave Foránea o Secundaria (Foreign Key - FK): Es una columna dentro de una tabla que hace referencia directa a la Llave Primaria de **otra tabla**. Su función es crear un vínculo o "relación" legal entre ambas. Por ejemplo, en una lista de Inscripciones, colocamos el carnet del alumno para saber a quién le pertenece ese registro sin tener que volver a escribir todo su nombre.

Implementación en Código SQL

Para trasladar nuestro diagrama al software, escribimos las instrucciones en lenguaje SQL especificando qué columna será nuestra Llave Primaria (`PRIMARY KEY`):

CREATE TABLE Alumnos (
    carnet INT PRIMARY KEY,
    nombre VARCHAR(50),
    carrera VARCHAR(50)
);

Con esta regla establecida, si intentamos ingresar por error a dos alumnos con el mismo número de carnet, el motor SQL bloqueará la acción inmediatamente protegiendo la integridad de la base de datos.

¿Por qué elegí este tema y dónde se aplica?

Mi perspectiva: Elegí este tema porque me llama mucho la atención cómo se ordena la información de manera lógica. Dibujar diagramas de flujo y luego estructurar las llaves primarias para asegurar que ningún registro se confunda o se pierda en el sistema es una de las partes más interesantes del desarrollo.

¿Dónde se aplica? Se aplica de forma directa en los sistemas de control de asignaciones universitarios, en los servidores de juegos que enlazan tu inventario con tu ID de usuario único, y en cualquier plataforma digital que deba guardar datos enlazados de forma segura.
